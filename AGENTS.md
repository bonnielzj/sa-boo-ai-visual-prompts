# Codex 工作规则

- 长任务、长文档、长日志、批量整理、图片/素材索引等内容，优先写入项目文件，不要把完整内容直接输出到聊天里。
- 聊天窗口只保留简短进度、关键判断、文件路径和下一步，避免触发上下文自动压缩失败。
- 需要保留过程证据时，创建清晰命名的 Markdown / JSON / CSV 文件，并在最终回复中引用路径。
- 如果任务会产生大量中间结果，先建立 `codex_outputs/` 或项目对应子目录，再分批写入。
- 新增或发现的长期可复用能力、skills、MCP 配置、工具使用经验、恢复摘要和关键工作规则，必须优先写入仓库内的持久化文件，再提醒用户推送到 GitHub。
- 不能只把重要上下文留在聊天里；如果当前仓库配置了 GitHub remote，并且用户明确允许推送，则在完成文件更新后执行 `git push`。
- 使用 `imagegen` 生成图片时，统一走 WE-AI AZ CLI/API 路径，不使用内置 `image_gen`。如果 Bonnie 明确说“快速输出”“快速预览”“我来判断是否 ok”等类似指令，优先走轻量预览流程：只做必要的提示词整理和一次 CLI/API 生图调用，生成后直接展示给 Bonnie 判断；除非 Bonnie 明确要求保存到项目、透明背景、精修、批量资产或可交付文件，否则不要额外做复杂后处理、多轮审查或长报告。
- 高优先级警告：凡是涉及 Bonnie 生图链路的“是否可用、默认走哪条路、是否已切换到指定 AZ 路由、是否命中 `/v1/images/generations` 或 `/v1/images/edits`”之类判断，必须先检查本机有效配置并做至少一次实际调用或等价可执行验证；未验证时不得口头假设、不得把内置 `image_gen` 说成默认主路由、不得把未确认的路由状态当成事实。此类误判会造成重大商业损失，按阻断级错误处理。
